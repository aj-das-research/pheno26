# Synthetic data dictionary (`synth_hpp/`)

Documents the columns and values in our synthetic tables, which mimic the HPP Pheno AI datasets used
in this project. Generated by `make_synthetic.py` (seed=42, 600 participants). On the HPP VM the real
tables share these column names/types (the synthetic mirrors them); the only differences are noted in
§"Synthetic vs real" at the end.

All tables are stored flat (parquet) and loaded by `pheno_io.get_df()` indexed on
`[participant_id, research_stage]`.

## Common columns (present in every table)

| Column | Type | Values |
|---|---|---|
| `participant_id` | int64 | 1000000–1000599 (600 unique participants) |
| `research_stage` | string | `00_00_visit` (baseline) and `01_00_call` (follow-up); `age_sex` is baseline-only |
| `cohort` | string | constant `10k` |
| `array_index` | int64 | constant `0` |
| `collection_date` | datetime64 | random dates in 2021–2022 (imaging/anthropometric tables) |
| `collection_timestamp` | float64 | all NaN (mirrors the real tables) |
| `timezone` | category | constant `asia/jerusalem` (imaging/anthropometric tables) |

A latent per-participant "metabolic risk" (0–1) drives the values so that adiposity phenotypes and
disease labels are correlated (the pipeline finds real signal).

## 1. `body_composition.parquet`  — the phenotypes (X)  · 695 rows × 159 cols · 498 participants

**Phenotype columns (152):**

- **`body_comp_<region>_<measure>` (145 cols, float64).**
  - regions (18): `android, gynoid, arm_left, arm_right, arms, arms_diff, leg_left, leg_right, legs, legs_diff, trunk, trunk_left, trunk_right, trunk_diff, total, total_left, total_right, total_diff`
  - measures (8): `bone_mass, fat_free_mass, fat_mass, lean_mass, region_percent_fat, tissue_mass, tissue_percent_fat, total_mass`
  - plus `body_comp_dose` (≈ 0.4)
  - values: mass/tissue measures are in **grams** (hundreds–thousands; `*_diff` columns can be **negative**, being left−right differences); `*_percent_fat` are **fractions ≈ 0.1–0.6**.
- **`total_scan_*` (7 cols, float64):** `sat_area, sat_mass, sat_volume` (subcutaneous), `vat_area, vat_mass, vat_volume` (visceral — these carry the strongest disease signal), `dose` (≈ 0.4).

Plus the common key/metadata columns. (Follow-up `01_00_call` rows are present with values; only the
disease tables NaN-out follow-up sub-fields.)

## 4. Disease label tables (D) — `curated_phenotypes/<disease>.parquet`

sub-fields, optional self-report booleans (with NaNs on follow-up rows), and the common keys.
`build_merged_data.py` binarizes the label per the `POSITIVE` rule.

## Coverage (mimics the real overlap)

`body_composition` is the **limiting cohort (498 unique participants)**; confounder/disease tables cover
~95–100% of those participants — matching the real HPP overlap, so a body-comp × disease merge keeps
~the full body-comp cohort.

## After `build_merged_data.py`

`MESHAgents/data/merged_data.csv` = baseline rows joined on `participant_id`:
the 152 `body_comp_*`/`total_scan_*` phenotypes (X) + `age, sex, height, weight, bmi,
waist_circumference, hip_circumference, waist_to_hip_ratio` (C) + one binary `<disease>` label (D).
(473 × 162 for `--disease diabetes`.)

## Synthetic vs real (what to expect on the VM)

- Column **names, dtypes, keys, label categories, and confounder sources match** the real HPP tables.
- The real **disease tables carry more sub-fields** than the few representative ones here — only
  `<disease>__curated_phenotype` + keys drive the pipeline, so this does not affect the code.
- Real value distributions differ (ours are synthetic Gaussians with an injected risk signal); the
  **ranges above are illustrative**, not clinically calibrated.
- Embedded `__sex`/`__age_at_research_stage` exist only in some real tables (e.g. osteoporosis); for the
  others, age/sex come from the `age_sex` table — handled by `build_merged_data.py`.
